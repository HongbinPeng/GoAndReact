- JWT 鉴权中间件

## 对用户模块实现的理解

### 1. 注册功能的实现思路

注册接口需要同时接收用户名、密码和头像文件，因此这里使用的是 `multipart/form-data`，而不是普通 JSON 请求。注册功能不仅仅是把数据写进数据库，还涉及真实文件上传场景的处理。

实现时重点关注了以下几点：

- 先读取 `username` 和 `password`，并判断是否为空。
- 再读取头像文件 `avatar`，因为题目要求注册时头像必传。
- 对头像做安全校验，包括文件大小不能超过 `10MB`、只允许常见图片扩展名、同时还要校验 MIME 类型，避免只改后缀名绕过限制。
- 文件保存时不能直接使用原文件名，否则不同用户上传同名文件可能会互相覆盖，因此需要生成唯一文件名。
- 密码不能明文存储，因此使用 `bcrypt` 进行加密后再写入数据库。

通过这个过程，可以更清楚地理解“表单字段”和“文件字段”在后端解析时的区别，也能认识到文件上传不仅是功能问题，更是安全问题。

### 2. 登录与 JWT 的实现思路

登录接口使用的是 `application/json`。实现思路如下：

- 根据用户名查询用户信息。
- 使用 `bcrypt` 比对用户输入的密码和数据库中的加密密码。
- 验证成功后生成 JWT。

JWT 中主要放入用户的 `id` 和 `username`，这样后续访问受保护接口时，不需要每次重新提交账号密码，只需要在请求头中携带：

```text
Authorization: Bearer <token>
```

这部分最大的收获在于，JWT 的核心价值不只是“登录成功后返回一个字符串”，而是把“用户身份”安全地传递给后续所有需要鉴权的接口。

### 3. 统一鉴权中间件的理解

为了避免在每个接口里重复解析 Token，项目中使用了统一的鉴权中间件。中间件的作用，就是把公共逻辑抽离出来，放在真正业务处理之前执行。

这个中间件主要做了三件事：

- 读取 `Authorization` 请求头
- 校验是否符合 `Bearer token` 格式
- 解析 JWT，并把当前用户信息写入 Gin 的上下文

这样一来，后续相册和照片模块只需要从上下文中取出当前用户，就能继续做权限判断，而不需要每个接口都重复编写 Token 解析逻辑。这也更能体现“工程化”的意义。

## 对相册模块鉴权实现的理解

相册模块看起来只是增删查改的一部分，但实际上权限控制才是核心。

### 1. 创建相册

创建相册时，前端只提交相册名、描述和是否公开，而真正的 `user_id` 不允许前端自己传，而是必须来自当前登录用户。这样设计的意义在于，前端不能伪造“替别人创建相册”的请求。

### 2. 获取当前用户相册列表

获取当前用户相册列表时，查询条件必须加上当前用户的 `user_id`。这里不能只写“查询 albums 表”，而是必须明确限制：

- 只能查询当前登录用户自己的相册

否则就会出现水平越权问题。

### 3. 获取公开广场相册列表

公开广场和“当前用户相册列表”最大的区别在于：

- 查询的是所有用户中 `is_public = true` 的相册
- 返回的数据里还要带上创建者的用户名和头像地址，方便前端展示

因此这里不仅是普通查询，还涉及相册和用户信息的组合返回。这个功能体现出后端接口设计不仅要考虑“查到数据”，还要考虑“前端需要什么结构的数据”。

## 对照片模块鉴权实现的理解

照片模块是最能体现权限控制价值的一部分。

### 1. 上传照片到指定相册

上传照片接口是：

```text
POST /api/v1/albums/:id/photos
```

这里最关键的问题不是“怎样保存图片”，而是“当前用户有没有资格往这个相册里传图”。

实现思路如下：

- 先根据路径参数里的相册 `id` 查询相册是否存在
- 再判断该相册的 `user_id` 是否等于当前登录用户的 `id`
- 只有相册创建者本人，才允许上传照片

在文件处理上，主要做了以下设计：

- 按相册 ID 进行目录划分，例如保存到 `uploads/photos/album_xxx/`
- 对图片再次做大小、扩展名和 MIME 校验
- 保存文件后，把 `album_id`、`file_path`、`file_size` 写入数据库

通过这个模块，可以更直观地看到“文件落盘”和“数据库记录”必须配合起来设计，不能只保存文件，不记录路径；也不能只写数据库，不真正保存文件。

### 2. 查看相册内照片列表

查看照片列表接口是：

```text
GET /api/v1/albums/:id/photos
```

这里的关键点在于：

- 如果相册是私有的，只有创建者本人可以查看
- 如果相册是公开的，登录用户都可以查看

因此处理逻辑是：

- 先查相册
- 再根据 `is_public` 和当前用户身份进行权限判断
- 权限通过后，再查询该相册下的照片列表

此外，还实现了：

- 按上传时间倒序返回
- 支持 `page` 和 `page_size` 分页参数

这个模块能够很好地说明：真正的权限控制，不是“接口能不能访问”，而是“访问到接口后，能不能访问到这条具体的数据”。

### 3. 访问图片资源的实现思路

题目里要求前端拿到 `avatar_url` 或 `file_path` 后，能够通过 URL 获取图片资源。这个需求表面上像是静态资源访问，但本质上也和权限控制有关。

在实现这个部分时，重点思考的是：

- 图片是否应该完全裸露为公开静态目录

如果直接暴露整个上传目录，虽然访问方便，但也可能绕过接口层的权限判断。因此在实现中更关注“如何在文件访问时继续做权限控制”的问题，也因此对后端资源访问安全有了更深的认识。

## 工程化与安全方面的收获

通过这次练习，对后端开发中的“工程化”和“安全意识”有了更具体的理解：

- 统一鉴权逻辑应该放到中间件，而不是散落在每个接口里。
- 密码必须加密存储，不能明文入库。
- 文件上传要同时校验扩展名、MIME 和大小。
- 权限校验不能只做表面校验，必须落实到具体资源上，例如某个相册、某张照片。
- 路由分组可以让接口结构更清晰，也更便于统一挂载中间件。
- 分页、错误码、统一返回结构这些细节，会直接影响接口是否规范、是否容易测试。

## 本次练习的总体收获

这次练习不再只是停留在“会写一个接口”的层面，而是开始从完整后端系统的角度思考问题。学习到的重点不只是 Gin 的用法，而是：

- 用户身份如何建立
- 身份如何通过 JWT 在多个接口间传递
- 权限如何针对不同资源进行校验
- 文件上传和数据库记录如何协同工作

本次练习最大的收获，就是更清楚地体会到：后端开发不仅是编写 CRUD，更重要的是对身份、权限、数据和资源访问关系的整体设计。
